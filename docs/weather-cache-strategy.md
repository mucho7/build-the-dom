# 날씨 캐시 전략

이 문서는 `우취될까?`가 KBO 직관 판단을 위해 날씨 데이터를 수집·보관·표시하는 방식을 기획자와 디자이너가 함께 논의하기 위한 기준 문서다.

## 1. 제품 목표

사용자는 경기와 날짜를 고른 뒤, 단순한 날씨 수치보다 **“오늘 직관을 가도 될까?”**에 대한 빠르고 직관적인 답을 받아야 한다.

이를 위해 사용자 화면에서 기상청 API를 실시간 호출하지 않는다. 서버가 미리 예보를 받아 계산한 뒤 DB에 보관하고, 화면은 준비된 결과를 읽는다.

## 2. 한눈에 보는 구조

```mermaid
flowchart LR
  A[GitHub Actions<br/>매시 17분] --> B[/api/cron/weather]
  B --> C[KBO 일정 동기화]
  B --> D[기상청 단기예보 수집]
  C --> E[(Game)]
  D --> F[(WeatherSnapshot)]
  G[ASOS 과거 관측 적재] --> H[(과거 관측값)]
  E --> I[유사 날씨 우천취소 통계 계산]
  H --> I
  D --> I
  I --> J[(HistoricalRainout)]
  K[사용자 날짜 선택] --> L[/api/weather?date=YYYYMMDD]
  E --> L
  F --> L
  J --> L
  L --> M[날짜별 경기 판단 결과]
```

## 3. 데이터 원천

| 목적 | 원천 | 사용 방식 |
| --- | --- | --- |
| 경기 일정·취소 상태 | KBO 공식 일정 | 월별 일정 동기화 |
| 앞으로의 경기 날씨 | 기상청 단기예보 | 경기 시간대와 경기 전 6시간 예보 추출 |
| 과거 날씨 | 기상청 ASOS 시간 관측 | 최근 3년 경기의 실제 강수량·경기 전 비 여부 적재 |

## 4. 캐시 범위와 갱신 주기

### 경기 예보 캐시

- 대상: 오늘·내일·모레의 예정 경기와 KBO가 진행·완료로 표기한 경기, 최대 15경기
- 갱신: GitHub Actions가 매시 17분에 `/api/cron/weather` 호출
- 저장 단위: 경기 1건당 최신 예보 스냅샷 1개 이상
- 실제 예보 기준 시각: 기상청 발표 시각
  - 현재 단기예보 발표 주기: 02·05·08·11·14·17·20·23시
  - 배치가 매시간 실행돼도 기상청의 새 발표본이 없으면 `예보 기준`은 바뀌지 않을 수 있음

### 사용자 응답 캐시

- 엔드포인트: `GET /api/weather?date=YYYYMMDD`
- 서버/CDN 캐시: 5분, 이후 최대 5분 동안 stale 응답 허용
- 브라우저 메모리: 날짜별 모든 경기 판단 결과를 한 번 받아 보관
- 효과: 같은 날짜 안에서 경기 탭을 바꿀 때 추가 API 요청 없이 즉시 전환

## 5. 저장 데이터

### Game

KBO 경기 날짜, 시작 시각, 홈·원정팀, 구장, 예정·완료·취소 상태를 저장한다.

### WeatherSnapshot

경기 판단에 필요한 예보 요약값을 저장한다.

| 필드 | 의미 |
| --- | --- |
| `issuedAt` | 기상청이 예보를 발표한 시각 |
| `forecastFor` | 경기 시작 시각 |
| `precipitationProbability` | 경기 시간대 강수확률 |
| `precipitationAmountMm` | 경기 시간대 예상 강수량 |
| `rainedBeforeGame` | 경기 전 6시간 이내 비 예상 여부 |
| `temperature`, `humidity` | 확장 가능 날씨 정보 |

현재는 시간별 원본 예보 전체를 저장하지 않고, 경기 판단에 필요한 대표 시간대와 경기 전 비 여부만 저장한다.

### HistoricalRainout

현재 예보 조건과 비슷한 최근 3년 경기의 우천취소 통계를 미리 계산해 저장한다.

| 필드 | 의미 |
| --- | --- |
| `similarGames` | 비교 대상 과거 경기 수 |
| `similarRainCancelledGames` | 그중 우천취소 경기 수 |
| `matchType` | `EXACT` 또는 `PRECIPITATION_ONLY` |
| `forecastIssuedAt` | 이 통계가 대응하는 최신 예보 발표 시각 |

## 6. 유사 날씨 판단 정책

### 1차: 완전 일치

아래 두 조건이 모두 같은 과거 경기를 찾는다.

- 예상 강수량 구간
  - 강수 없음
  - 1mm 미만
  - 1~4.9mm
  - 5mm 이상
- 경기 전 비 여부

표본이 5경기 이상이면 위험도 score와 참고 문구에 모두 사용한다.

### 2차: 강수량 보조값

완전 일치 표본이 0건일 때만 강수량 구간이 같은 과거 경기로 범위를 넓힌다.

- 화면 문구에는 표시
- 표본이 5경기 이상이고 우천취소율이 60% 이상일 때만 score 하한에 반영
- “경기 전 비 조건은 제외한 보조 참고”임을 명시

이 정책은 표본이 너무 적어 아무 정보도 보여주지 못하는 상황을 줄이기 위한 장치다.

## 7. 위험도 score 정책

기본 점수는 현재 예보로 계산한다.

| 조건 | 점수 |
| --- | --- |
| 강수확률 20 / 40 / 70% 이상 | +10 / +22 / +40 |
| 예상 강수량 0초과 / 2 / 5mm 이상 | +8 / +20 / +35 |
| 경기 전 비 예상 | +12 |

점수 구간은 다음과 같다.

| score | 사용자 판단 |
| --- | --- |
| 0~14 | 우취 걱정 낮음 |
| 15~37 | 우산은 챙겨요 |
| 38~64 | 출발 전 확인 필요 |
| 65 이상 | 우취 걱정 높음 |

돔구장은 비와 관계없이 `강수 영향 낮음`으로 처리한다.

완전 일치 표본이 5경기 이상이면, 최근 3년 우천취소율을 score의 하한으로 적용한다. 강수량만 일치하는 보조 표본도 5경기 이상이고 우천취소율이 60% 이상이면 같은 방식으로 반영한다. 이력 표본을 현재 예보보다 우선하는 강한 신호로 취급하기 위한 정책이다.

```
최종 score = max(현재 예보 score, 조건을 충족한 유사 표본의 우천취소율 × 100)
```

예를 들어 비슷한 조건의 6경기 중 5경기가 우천취소됐다면, 현재 예보 score가 낮더라도 최종 score는 최소 83점이 되어 `우취 걱정 높음`으로 표시된다. 강수량만 일치한 표본도 취소율이 60% 이상이면 이 규칙을 적용한다.

## 8. 사용자 화면에서 보이는 정보

판단 카드는 두 영역으로 구성된다.

### 기본 영역

- 경기 날짜·시작 시각·대진·등록된 구장 풀네임
- 기상청 예보 발표 기준 시각
- 우취 판단 상태와 헤드라인

이 영역은 카드 높이를 고정해 날짜·경기 전환 중 레이아웃이 흔들리지 않게 한다.

### 상세 영역

`ChevronDown` 아이콘 버튼을 누르면 카드 안에서 펼쳐진다. 다음 항목을 표로 제공한다.

- 경기 전 비 예상 여부
- 경기 중 강수확률과 예상 강수량
- 최근 3년 유사 경기 수
- 유사 경기의 우천취소·진행 건수
- 현재 판단 근거와 준비 팁

KBO가 경기를 `played`로 표시하면 날씨 판단 대신 팀별 KBO 스코어를 카드에 표시한다.

### 표현 원칙

- `예보 기준`은 기상청 발표 시각이다. 서비스가 마지막으로 캐시를 갱신한 시각과 다를 수 있다.
- 이력 통계는 확률을 단정하는 근거가 아니라 참고 정보다.
- 우천취소 확정 여부는 KBO 공식 공지가 최종 기준이다.

## 9. 정상·대기·오류 상태

| 상태 | 사용자 메시지 | 의미 |
| --- | --- | --- |
| 정상 | 판단 카드 표시 | 최신 예보 캐시 존재 |
| 캐시 대기 | 최신 예보 캐시를 준비하고 있어요 | 배치가 아직 해당 경기 예보를 저장하지 않음 |
| 통계 없음 | 상세 표에 `표본 없음` 표시 | 유사 조건 표본이 없음 또는 이력 적재 전 |
| 예보 오류 | 아직 판단하기 어려워요 | 캐시·DB·외부 데이터 처리 오류 |
| 경기 없음 | 이 날짜에는 예정된 경기가 없어요 | KBO 예정 경기 없음 |

## 10. 현재 제약과 향후 고도화 후보

### 현재 제약

- 시간별 예보 원본은 DB에 남기지 않는다.
- 과거 관측값은 실제 날씨이며, 당시 기상청 예보를 완전히 복원한 데이터는 아니다.
- GitHub Actions 예약 실행은 지연될 수 있다.
- `예보 기준`만 노출되어 캐시 갱신이 정상인지 사용자가 즉시 알기 어렵다.

### 고도화 후보

1. 경기 전후 시간별 예보 타임라인 제공
2. `기상청 발표 시각`과 `서비스 캐시 갱신 시각` 분리 표시
3. 캐시 갱신 실패 알림과 마지막 정상 캐시 유지
4. 유사 조건의 경기 목록·취소 사유를 상세 화면으로 제공
5. 응원팀·관심 구장 알림

## 11. 기획·디자인 논의 질문

1. 사용자는 score 숫자보다 상태 문구를 더 신뢰할까, 둘 다 필요할까?
2. 보조 통계는 기본 카드에 바로 보여줄까, “왜?”를 눌렀을 때만 보여줄까?
3. `예보 기준` 외에 `마지막 갱신`을 어느 정도 강조할까?
4. 강수확률·강수량·경기 전 비 중 어떤 항목을 카드에서 가장 앞에 둘까?
5. 시간별 예보 타임라인은 MVP 이후 얼마나 우선순위가 높을까?
6. 우천취소 확정 경기와 단순 위험 예보의 시각적 구분은 어느 정도 강해야 할까?
